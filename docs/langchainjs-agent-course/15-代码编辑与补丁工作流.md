# 15. 代码编辑与补丁工作流：让 Agent 安全修改项目

## 本章目标

第 14 章建立了高风险操作审批。本章把审批能力用于代码修改：Agent 先生成补丁计划，用户审查后再应用。

企业级开发 Agent 不能只读文件和运行命令，还要能修改代码。但直接让模型写文件很危险。

本章实现“补丁优先”的编辑工作流：

```text
读取文件 → 生成 patch → 用户审查 → 应用 patch → 运行测试
```

新增模块：

```text
src/editing/patch-schema.ts
src/editing/patch-planner.ts
src/tools/apply-patch.ts
```

## 1. 为什么不要直接写文件

直接写文件的问题：

- 容易覆盖用户改动。
- 难以审查变更。
- 出错后不易回滚。
- 审计日志不清楚。

补丁工作流更适合企业：

- 先展示 diff。
- 用户确认后再应用。
- 可以配合测试和回滚。

## 2. Patch 结构

创建 `src/editing/patch-schema.ts`：

```ts
import { z } from "zod";

export const FilePatchSchema = z.object({
  path: z.string(),
  reason: z.string(),
  beforeSummary: z.string(),
  afterSummary: z.string(),
  unifiedDiff: z.string(),
});

export const PatchPlanSchema = z.object({
  task: z.string(),
  risk: z.enum(["low", "medium", "high"]),
  patches: z.array(FilePatchSchema),
  testsToRun: z.array(z.string()),
});
```

## 3. 生成补丁计划

`patch-planner` 不直接修改文件，只生成变更计划：

```ts
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { createChatModel } from "../models/chat.js";
import { PatchPlanSchema } from "./patch-schema.js";

const prompt = ChatPromptTemplate.fromMessages([
  ["system", `
你是代码修改规划器。
只能输出补丁计划，不要声称已修改文件。
补丁必须最小化，避免无关改动。
`.trim()],
  ["human", "任务：{task}\n相关文件：\n{context}"],
]);

export async function createPatchPlan(task: string, context: string) {
  return prompt.pipe(createChatModel().withStructuredOutput(PatchPlanSchema)).invoke({ task, context });
}
```

## 4. apply_patch 工具

第一版可以不让模型直接调用写文件工具，而是让 CLI 展示 patch 后由用户确认。

```text
mini-agent edit "给 ask 命令增加 --json 输出"
```

流程：

1. Agent 搜索相关文件。
2. 生成 patch plan。
3. CLI 打印 diff。
4. 用户确认。
5. 应用 patch。
6. 运行 typecheck/test。

## 5. 安全规则

代码编辑必须遵守：

- 默认只改工作区内文件。
- 不改 `.env`、密钥、锁文件，除非明确请求。
- 不删除用户文件。
- 不做大规模重构，除非计划中明确说明。
- 应用 patch 前必须展示 diff。
- 应用后必须建议或执行验证命令。

## 6. 验收

```bash
mini-agent edit "给 cli 添加 version 命令说明"
```

期望：

- 输出 patch plan。
- 展示将修改的文件。
- 要求确认。
- 应用后提示运行 `npm run typecheck`。
