# 22. 最终企业架构蓝图

## 本章目标

前面 01–21 章分别实现和增强了各个模块。本章把它们收束成一张企业 Agent 平台蓝图。

本章把整个 `mini-agent-langchain` 从教学项目提升为企业 Agent 平台蓝图。

## 1. 最终能力地图

```text
用户入口
 ├─ CLI
 ├─ HTTP API
 ├─ Web UI
 └─ 企业 IM

Agent Runtime
 ├─ ask chain
 ├─ task agent
 ├─ planned graph
 ├─ approval graph
 └─ multi-agent graph

Capabilities
 ├─ filesystem tools
 ├─ command tools

 ├─ code editing tools
 ├─ business plugins
 └─ external APIs

Enterprise Layer
 ├─ authn/authz

 ├─ audit log

 ├─ evals
 ├─ model router
 ├─ cost control
 └─ deployment
```

## 2. 分层架构

### Interface Layer

负责输入输出：

- CLI。
- HTTP API。
- WebSocket / SSE。
- 企业 IM Bot。

### Application Layer

负责编排任务：

- `askService`。
- `taskService`。
- `chatService`。
- `indexService`。
- `evalService`。

### Agent Runtime Layer

负责智能体执行：

- LangChain `createAgent()`。
- LangGraph 状态机。
- 多 Agent 协作。
- Human-in-the-loop。

### Capability Layer

负责工具能力：

- 文件。
- 命令。
- RAG。
- 代码编辑。
- 插件。

### Governance Layer

负责企业治理：

- 权限。
- 审批。
- 审计。
- 成本。
- 评估。
- 安全。

## 3. 推荐里程碑

```text
M1: CLI MVP
  ask / run / tools

M2: 可控 Agent
  LangGraph / memory / approval

M3: 知识库
  RAG / 引用 / 增量索引

M4: 开发助手
  命令执行 / 代码编辑 / 测试分析

M5: 企业化
  API / 权限 / 审计 / 评估 / 部署

M6: 平台化
  插件 / 多 Agent / 多模型 / 多入口
```

## 4. 最终验收任务

```bash
mini-agent run --team "
阅读当前项目，判断它距离企业级 Agent 平台还缺哪些能力。
请按安全、工程化、知识库、工具生态、部署五类输出，并给出引用依据。
"
```

合格输出应包含：

- 使用了文件和文档检索工具。
- 有来源引用。
- 有风险分级。
- 有可执行改进计划。
- 没有编造项目中不存在的能力。
