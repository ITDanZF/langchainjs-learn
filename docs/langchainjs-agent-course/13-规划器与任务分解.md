# 13. 规划器与任务分解：从单步 Agent 到复杂任务执行

## 本章目标

第 12 章已经完成基础闭环：CLI、模型、工具、Agent、Graph、记忆、RAG 和评估。

从本章开始进入企业增强篇：不重写基础能力，而是在第 06 章的 Graph、第 09 章的工具箱、第 11 章的评估体系之上继续升级。

前面的 `run` 命令已经能调用工具，但它仍然偏“边想边做”。企业级 Agent 面对复杂任务时，应该先规划、再执行、最后复盘。

本章会新增：

```text

```

完成后支持：

```bash
mini-agent plan "分析当前项目架构，并提出三条改进建议"
mini-agent run --planned "分析当前项目架构，并提出三条改进建议"
```

## 1. 为什么需要规划器

没有规划器时，Agent 容易：

- 一上来就读错文件。
- 工具调用顺序混乱。
- 长任务中途丢目标。
- 无法向用户解释执行路径。

规划器的作用是先生成任务蓝图：

```text
目标 → 步骤 → 所需工具 → 风险 → 验收标准
```

## 2. 规划数据结构

创建 `src/planning/plan-schema.ts`：

```ts
import { z } from "zod";

export const PlanStepSchema = z.object({
  id: z.string(),
  title: z.string(),
  goal: z.string(),
  suggestedTools: z.array(z.string()),
  status: z.enum(["pending", "running", "completed", "failed"]).default("pending"),
});

export const TaskPlanSchema = z.object({
  task: z.string(),
  summary: z.string(),
  risk: z.enum(["low", "medium", "high"]),
  steps: z.array(PlanStepSchema),
  doneCriteria: z.array(z.string()),
});

export type TaskPlan = z.infer<typeof TaskPlanSchema>;
```

## 3. 规划器 Chain

创建 `src/planning/planner.ts`：

```ts
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { createChatModel } from "../models/chat.js";
import { TaskPlanSchema } from "./plan-schema.js";

const prompt = ChatPromptTemplate.fromMessages([
  ["system", `
你是企业级 Agent 的任务规划器。
你只负责规划，不执行工具。
输出必须符合结构化 schema。
规划要具体、可执行、步骤数量控制在 3 到 7 步。
`.trim()],
  ["human", "任务：{task}"],
]);

export async function createTaskPlan(task: string) {
  const model = createChatModel().withStructuredOutput(TaskPlanSchema);
  const chain = prompt.pipe(model);
  return chain.invoke({ task });
}
```

## 4. CLI 增加 plan 命令

在 `src/cli.ts` 中添加：

```ts
import { createTaskPlan } from "./planning/planner.js";

program
  .command("plan")
  .description("Create a task plan without executing tools")
  .argument("<task...>", "task text")
  .action(async (task: string[]) => {
    const input = joinArgs(task);
    if (!ensureInput(input, "请输入任务")) return;

    const plan = await createTaskPlan(input);
    console.log(JSON.stringify(plan, null, 2));
  });
```

## 5. 和旧主线怎么衔接

规划器不会替代原来的 `runGraphTask()`，而是给它增加前置步骤：

```text
用户任务
  ↓
第 13 章 planner 生成结构化计划
  ↓
第 06 章 task graph 按计划执行工具
  ↓
第 11 章 eval 检查结果质量
```

所以 `run --planned` 可以先复用已有 `run --graph`，等后续第 18 章再拆成多 Agent。

## 6. 规划 + 执行 Graph

规划型 Agent 的图可以是：

```text
START
  ↓
planner
  ↓
executor
  ↓
reviewer
  ↓
END
```

创建 `src/graph/planned-task-graph.ts` 时，可以把状态扩展为：

```ts
const PlannedTaskState = Annotation.Root({
  task: Annotation<string>(),
  plan: Annotation<TaskPlan | undefined>(),
  messages: Annotation<BaseMessage[]>({
    reducer: (current, update) => current.concat(update),
    default: () => [],
  }),
  finalAnswer: Annotation<string | undefined>(),
});
```

第一版可以先做到：生成计划 → 把计划和任务一起交给 `runGraphTask()` 执行。

## 7. 规划质量要求

一个好的计划应该：

- 不超过 7 步。
- 每步都有明确目标。
- 工具建议具体，例如 `list_files`、`read_file`。
- 高风险任务标记为 `medium` 或 `high`。
- 有明确验收标准。

## 8. 验收

```bash
mini-agent plan "检查当前项目是否具备发布为 npm CLI 的条件"
```

输出应该包含：

- 检查 `package.json`。
- 检查 `bin` 配置。
- 检查 build 脚本。
- 检查 README 或使用说明。
- 给出完成标准。

## 9. 企业级思考

规划器是多 Agent 系统的入口。未来可以拆成：

- Planner：拆任务。
- Executor：执行工具。
- Reviewer：审查结果。
- Reporter：生成交付报告。

下一章会实现更关键的企业能力：Human-in-the-loop 审批。
