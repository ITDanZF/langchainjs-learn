# 00. 课程总览：让 Agent CLI 一步步长出来

## 本课程的主线

你已经创建了 `mini-agent-langchain` 文件夹。本课程会直接以它作为项目根目录，从零开始构建。

但“从零开始”不等于第一章就创建完整企业目录。我们会按真实项目的节奏推进：

```text
先能运行
再接模型
再改善输出
再加工具
再让 Agent 使用工具
再把运行过程变得可控
再补记忆、RAG、权限、工程化和评估
```

每一步都要能运行、能验收、能解释为什么现在需要它。

## 为什么从 CLI 开始

CLI 是学习 Agent 的最佳入口：

- 输入输出简单，容易调试。
- 能自然接入文件、搜索、命令等工具。
- 很适合模拟真实开发助手。
- 后续可以复用核心模块接入 Web、服务端 API 或企业 IM。

企业级 Agent 不等于一开始就复杂，而是每一步都保持边界清晰、可测试、可替换。

## 先看主线导图

如果你担心章节多、功能散，先看 [主线导图](./ROADMAP.md)。它说明每一章新增什么，以及为什么不提前搭完整架构。

## 能力演进路线

### 阶段一：最小 CLI

产物：

```bash
npm run dev -- ask "你好"
```

这一阶段只做命令行入口和参数处理。不会接模型，不会创建工具目录，也不会提前写 `run`、`chat` 占位命令。

### 阶段二：模型问答

产物：

```bash
npm run dev -- ask "用一句话解释 LangChain.js Agent"
```

这时才引入：

- `.env` 和配置校验。
- ChatModel 封装。
- Prompt 模板。
- Ask Chain。

### 阶段三：流式输出

产物：

```bash
npm run dev -- ask "给我一个学习计划"
npm run dev -- ask --no-stream "给我一个学习计划"
```

这一阶段只解决体验问题：模型生成一点，终端显示一点。

### 阶段四：本地工具

产物：

```bash
npm run dev -- run "列出当前项目文件"
```

注意节奏：先实现工具模块，再在下一章交给 Agent。工具系统需要先明确工作区边界、输入校验和输出限制。

### 阶段五：工具 Agent

产物：

```bash
npm run dev -- run "阅读 README.md 并总结项目"
```

这时项目第一次从“问答程序”变成“任务执行程序”。模型可以判断是否需要工具，并基于工具结果回答。

### 阶段六：可控运行时

产物：

```bash
npm run dev -- run --graph "搜索 docs 中和 RAG 有关的章节"
```

这时才引入 LangGraph。原因不是“架构应该高级”，而是项目已经需要显式控制循环、分支、权限、中断和恢复。

### 阶段七：记忆与知识库

产物：

```bash
npm run dev -- chat --thread project-a
npm run dev -- index docs
```

这一阶段在已有 Graph 和工具系统上继续加能力，而不是另起一套项目。

### 阶段八：工程化和企业增强

产物：

```bash
npm run dev -- eval
npm run test
npm run build
```

当系统已经可用后，再补：

- 日志与错误处理。
- 权限与审计。
- 评估与观测。
- 规划、审批、代码编辑、多 Agent、服务化、插件化。

## 每章固定节奏

每章都按同一个模板推进：

1. 当前项目已经有什么。
2. 现在遇到什么新问题。
3. 本章只新增什么。
4. 哪些能力明确留到后续章节。
5. 如何运行验收。

这样读者看到的不是一张终局蓝图，而是一条真实的迭代路径。

## 暂不提前创建的东西

第 01 章不会创建这些目录：

```text
src/models/
src/tools/
src/agents/
src/graph/
src/memory/
src/rag/
src/evals/
src/observability/
tests/
```

它们会在第一次真正需要时出现。这个约束很重要，因为它能让每个目录都有明确来历。

## 终局蓝图放在哪里

最终企业架构不是消失了，而是放到第 22 章作为回看和升级方向。

到那时你已经亲手走过：

```text
CLI → Model → Stream → Tools → Agent → Graph → Memory → RAG → Eval → Platform
```

再看完整架构，才不会变成抽象名词堆叠。

## 每章验收标准

每章结束都必须满足：

- 项目能运行。
- 新增能力有明确命令可验收。
- 只引入当前需要的依赖。
- 只创建当前需要的目录。
- 不为了赶进度破坏安全边界。

下一章开始，我们只做一件事：把 `mini-agent-langchain` 变成一个能接收 `ask` 命令的最小 TypeScript CLI。
