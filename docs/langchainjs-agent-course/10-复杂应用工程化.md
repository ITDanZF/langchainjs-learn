## 1. 企业级不是功能堆叠

一个 Agent CLI 能跑起来只是第一步。企业项目还需要：

- 配置清晰。
- 日志可追踪。
- 错误可理解。
- 模块边界稳定。
- 测试能覆盖关键逻辑。
- 安全策略可审计。

本章不增加新 Agent 能力，而是让已有能力更可靠。

## 2. 本地状态目录边界

第 02 章已经引入 `MINI_AGENT_HOME` 和 `ensureAgentHome()`。从工程化角度看，这个目录就是 mini-agent 自己的“用户级工作目录”，类似 Codex 的 `.codex`：

```text
~/.mini-agent/
  config.toml
  auth.json
  history.jsonl
  sessions/
  logs/
  cache/
  skills/
  plugins/
```

推荐约定：

| 内容 | 放在哪里 | 原因 |
| --- | --- | --- |
| 全局配置 | `~/.mini-agent/config.toml` | 跨项目复用 |
| 认证信息 | `~/.mini-agent/auth.json` | 不进入项目 Git |
| 命令历史 | `~/.mini-agent/history.jsonl` | 方便统一查询 |
| 多轮会话 | `~/.mini-agent/sessions/` | 支持 resume 和归档 |
| 运行日志 | `~/.mini-agent/logs/` | 与源码分离 |
| 缓存数据 | `~/.mini-agent/cache/` | 可删除、可重建 |
| 当前项目配置 | `<project>/.mini-agent/config.toml` | 只覆盖当前项目 |

`AGENT_WORKSPACE` 和 `MINI_AGENT_HOME` 不要混用：

- `AGENT_WORKSPACE` 是 Agent 被允许观察和操作的项目目录。
- `MINI_AGENT_HOME` 是 CLI 自己保存状态的目录。

后续所有历史、会话、日志、缓存能力，都应该优先写入 `MINI_AGENT_HOME`，而不是写到 `src/`、项目根目录或用户当前命令行目录。

## 3. 推荐模块边界

当前项目建议保持：

```text
src/
  main.ts                # 只做命令解析和输出
  config/                # 环境变量、运行配置、本地状态目录
  models/                # 模型供应商封装
  prompts/               # 系统提示词和模板
  chains/                # 简单 Runnable 链
  tools/                 # 工具定义和策略
  agents/                # createAgent 运行时
  graph/                 # LangGraph 运行时
  memory/                # thread 和 checkpointer
  rag/                   # 文档加载、切分、索引、检索
  observability/         # 日志、trace、metrics
  utils/                 # 通用纯工具函数
```

原则：`main.ts` 不直接操作模型、文件系统、向量库或权限细节。

## 4. 日志模块

创建 `src/observability/logger.ts`：

```ts
import { env } from "../config/env.js";

const levels = ["debug", "info", "warn", "error"] as const;
type Level = (typeof levels)[number];

function shouldLog(level: Level) {
  return levels.indexOf(level) >= levels.indexOf(env.LOG_LEVEL);
}

export const logger = {
  debug(message: string, data?: unknown) {
    if (shouldLog("debug")) console.error(format("debug", message, data));
  },
  info(message: string, data?: unknown) {
    if (shouldLog("info")) console.error(format("info", message, data));
  },
  warn(message: string, data?: unknown) {
    if (shouldLog("warn")) console.error(format("warn", message, data));
  },
  error(message: string, data?: unknown) {
    if (shouldLog("error")) console.error(format("error", message, data));
  },
};

function format(level: Level, message: string, data?: unknown) {
  const payload = data === undefined ? "" : ` ${JSON.stringify(data)}`;
  return `[${new Date().toISOString()}] ${level.toUpperCase()} ${message}${payload}`;
}
```

日志写到 `stderr`，避免污染 `stdout`。这对 CLI 很重要，因为用户可能把标准输出重定向到文件。

## 5. 错误类型

创建 `src/utils/errors.ts`：

```ts
export class AppError extends Error {
  constructor(
    message: string,
    public readonly code: string,
    public readonly exitCode = 1,
  ) {
    super(message);
  }
}

export function getErrorMessage(error: unknown) {
  if (error instanceof Error) return error.message;
  return String(error);
}
```

CLI 顶层可以统一捕获：

```ts
try {
  await program.parseAsync(process.argv);
} catch (error) {
  logger.error("Command failed", { error: getErrorMessage(error) });
  console.error(getErrorMessage(error));
  process.exitCode = error instanceof AppError ? error.exitCode : 1;
}
```

## 6. 工具策略测试

创建 `tests/command-policy.test.ts`：

```ts
import { describe, expect, it } from "vitest";
import { validateCommand } from "../src/tools/command-policy.js";

describe("command policy", () => {
  it("allows low-risk commands", () => {
    expect(() => validateCommand("pwd", [])).not.toThrow();
    expect(() => validateCommand("npm", ["test"])).not.toThrow();
  });

  it("blocks dangerous commands", () => {
    expect(() => validateCommand("rm", ["-rf", "."])).toThrow();
    expect(() => validateCommand("git", ["reset", "--hard"])).toThrow();
  });
});
```

运行：

```bash
npm run test
```

安全策略必须有测试，因为它是企业 Agent 的底线。
