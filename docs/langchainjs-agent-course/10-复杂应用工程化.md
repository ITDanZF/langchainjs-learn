# 10. 复杂应用工程化

## 本章目标

当 LangChain.js 应用从 demo 走向真实项目时，最重要的问题不再只是“能不能调用模型”，而是：

- 模块是否清晰。
- 配置是否安全。
- 错误是否可处理。
- 日志是否可追踪。
- 工具是否可测试。
- 成本是否可控。

本章会整理一个复杂应用的工程化结构。

## 1. 推荐目录结构

```text
src/
  config.ts
  index.ts
  models/
    chat.ts
    embeddings.ts
  prompts/
    templates.ts
    system.ts
  chains/
    basic-chain.ts
    rag-chain.ts
  tools/
    index.ts
    read-file.ts
    list-files.ts
    search-text.ts
    run-command.ts
    business-api.ts
  agents/
    react-agent.ts
    graph-agent.ts
    business-agent.ts
  rag/
    loader.ts
    splitter.ts
    retriever.ts
  memory/
    checkpointer.ts
  logging/
    logger.ts
  evals/
    cases.ts
    run-eval.ts
```

核心原则是：

```text
模型、Prompt、工具、Agent、RAG、记忆、评估分开管理
```

## 2. 配置管理

`src/config.ts` 应该集中处理环境变量：

```ts
import "dotenv/config";
import { z } from "zod";

const envSchema = z.object({
  DEEPSEEK_API_KEY: z.string().min(1),
  DEEPSEEK_MODEL: z.string().default("deepseek-chat"),
  LANGSMITH_TRACING: z.string().optional(),
  LANGSMITH_API_KEY: z.string().optional(),
});

export const config = envSchema.parse(process.env);
```

这样可以在启动时尽早发现配置缺失。

## 3. 模型模块

`src/models/chat.ts`：

```ts
import { ChatOpenAI } from "@langchain/openai";
import { config } from "../config";

export const chatModel = new ChatOpenAI({
  model: config.DEEPSEEK_MODEL,
  apiKey: config.DEEPSEEK_API_KEY,
  configuration: {
    baseURL: "https://api.deepseek.com",
  },
});
```

不要在业务代码里到处 new 模型实例。

统一模型入口更方便：

- 切换供应商。
- 控制温度。
- 加 tracing。
- 做重试。
- 做测试替身。

## 4. 工具模块边界

每个工具单独一个文件：

```text
tools/read-file.ts
tools/search-text.ts
tools/business-api.ts
```

`tools/index.ts` 只做汇总：

```ts
export const tools = [
  readFileTool,
  searchTextTool,
  queryTicketTool,
];
```

工具函数内部要尽量小而清晰，避免把复杂业务流程塞进一个工具。

## 5. 错误处理策略

常见错误分三类：

- 用户输入错误：缺少参数、路径不存在。
- 工具执行错误：文件权限、API 超时。
- 模型调用错误：限流、鉴权失败、网络错误。

对 agent 来说，很多工具错误应该返回给模型，而不是直接让程序崩溃。

例如：

```ts
return `工具执行失败：${message}`;
```

但配置错误、API Key 缺失这类启动错误，应该直接终止。

## 6. 日志与调试

最少应该记录：

- 用户输入。
- 调用的 agent 或 chain。
- 工具名称。
- 工具参数。
- 工具耗时。
- 模型耗时。
- 最终输出摘要。

不要在日志里输出 API Key、用户隐私或完整敏感文件内容。

## 7. 测试策略

复杂 LangChain.js 应用至少需要三类测试：

- 工具单元测试：工具输入输出是否稳定。
- Chain 测试：Prompt 和 parser 是否能正常组合。
- 回归评估：典型问题的回答质量是否退化。

工具最好先脱离模型单独测试。

模型相关测试要控制成本，可以使用小样本和 Mock。

## 8. 本章验收

完成后，你应该能：

- 设计清晰的 LangChain.js 项目结构。
- 集中管理配置。
- 拆分模型、工具、agent 和 RAG 模块。
- 为工具和模型调用设计错误处理。
- 制定基础日志和测试策略。

下一章会学习评估、观测和部署。
