# 03b. 工作目录与本地状态目录

## 本章目标

第 04 章会开始实现 `list_files`、`read_file`、`search_text` 这类文件工具。在写工具之前，必须先定义两个边界：

```text
workspaceRoot      # Agent 可以观察和操作的工作目录
agentHome          # mini-agent 自己保存配置和状态的本地目录
```

本章要完成三件事：

```text
1. 如果没有提供 workspaceRoot，默认使用用户目录下的 workspaceRoot。
2. 完善 createHomeRoot()，负责创建 agentHome 和默认 config.json。
3. 后续程序主要从 config.json 读取配置，而不是继续依赖 .env。
```

完成后重点更新：

```text
src/workspace/path.ts
src/config/index.ts
```

`.env` 以后只适合作为启动时的少量覆盖项，例如 `MINI_AGENT_HOME` 或临时密钥；长期配置应该进入 `agentHome/config.json`。如果关键配置为空，程序启动时应该进入交互式输入流程，引导用户补齐配置。

## 1. 为什么要先定义工作目录

文件工具一旦交给模型使用，就不能让它随意读取整台电脑。

如果没有工作目录边界，模型可能尝试读取：

```text
C:\Users\<你>\.ssh
C:\Users\<你>\.codex\auth.json
..\..\其他项目\.env
```

所以在实现任何文件工具之前，先规定：

```text
所有文件访问都必须限制在 workspaceRoot 内部。
```

这不是为了让代码复杂，而是为了给工具系统建立第一道安全边界。

## 2. 两个目录不要混用

`mini-agent-langchain` 会同时有两个重要目录：

| 目录 | 用途 | 示例 |
| --- | --- | --- |
| `workspaceRoot` | Agent 当前处理的项目目录 | `C:\Users\<你>\workspaceRoot` |
| `agentHome` | mini-agent 自己的本地状态目录 | `C:\Users\<你>\.mini-agent` |

可以这样理解：

```text
workspaceRoot
= Agent 要工作的地方
= 文件工具能访问的范围
= read_file / list_files / search_text 的安全边界

agentHome
= mini-agent 自己的家
= 保存 config.json、历史、会话、日志、缓存
= 类似 Codex 的 ~/.codex
```

第 04 章的文件工具只应该访问 `workspaceRoot`，不应该读取 `agentHome` 里的认证、历史或缓存。

## 3. 本地状态目录如何生成

`agentHome` 不是系统自动生成的，也不是 Git 仓库生成的，而是程序根据用户主目录拼出来的。

Node.js 中可以这样得到用户主目录：

```ts
import { homedir } from "node:os";
import path from "node:path";

const defaultAgentHome = path.join(homedir(), ".mini-agent");
```

在 Windows 上，`homedir()` 通常返回：

```text
C:\Users\Administrator
```

所以 `path.join(homedir(), ".mini-agent")` 会得到：

```text
C:\Users\Administrator\.mini-agent
```

在 macOS / Linux 上会得到：

```text
/Users/<你>/.mini-agent
/home/<你>/.mini-agent
```

推荐规则是：默认初始化，同时允许用户修改。

```text
默认值：
  path.join(homedir(), ".mini-agent")

用户覆盖：
  MINI_AGENT_HOME 环境变量

未来可选：
  --home 命令行参数
```

也就是：

```ts
export function getAgentHome() {
  return process.env.MINI_AGENT_HOME || path.join(homedir(), ".mini-agent");
}
```

如果用户想把状态目录放到其他磁盘，可以临时设置：

```powershell
$env:MINI_AGENT_HOME="D:\mini-agent-home"
npm run dev
```

这个设计和 Codex 的 `.codex` 类似：普通用户不用配置，高级用户可以迁移到自己想要的位置。

## 4. createHomeRoot 的职责

`createHomeRoot()` 是本章最重要的初始化函数。它不只是创建目录，还要创建默认配置文件。

它应该负责：

```text
1. 计算 agentHome。
2. 创建 agentHome 目录。
3. 检查 agentHome/config.json 是否存在。
4. 如果不存在，就写入默认配置。
5. 返回 agentHome 和 config.json 的路径，方便后续读取。
```

默认生成的目录结构可以是：

```text
C:\Users\<你>\.mini-agent
  config.json
  logs\
  sessions\
```

其中 `config.json` 是之后程序读取配置的主要来源。

推荐的默认配置内容：

```json
{
  "MODEL_PROVIDER": "",
  "MODEL_NAME": "",
  "MODEL_BASE_URL": "",
  "MODEL_API_KEY": "",
  "AGENT_WORKSPACE": "",
  "LOG_LEVEL": "info"
}
```

这里故意让模型配置和 `AGENT_WORKSPACE` 都默认为空字符串。它的含义是：

```text
MODEL_PROVIDER / MODEL_NAME / MODEL_BASE_URL / MODEL_API_KEY 为空：
  需要在程序启动时提示用户输入。

AGENT_WORKSPACE 为空：
  没有显式配置 workspaceRoot，需要使用默认工作目录。
```

也就是说，`config.json` 不应该绑定某一个具体模型厂商。DeepSeek、OpenAI、本地模型、其他 OpenAI-compatible 服务，都可以通过同一组字段表达。

示例代码：

```ts
import { mkdir, writeFile } from "node:fs/promises";
import { existsSync } from "node:fs";
import { homedir } from "node:os";
import path from "node:path";

export function getAgentHome() {
  return process.env.MINI_AGENT_HOME || path.join(homedir(), ".mini-agent");
}

export async function createHomeRoot() {
  const agentHome = getAgentHome();
  const configPath = path.join(agentHome, "config.json");

  await mkdir(agentHome, { recursive: true });
  await mkdir(path.join(agentHome, "logs"), { recursive: true });
  await mkdir(path.join(agentHome, "sessions"), { recursive: true });

  if (!existsSync(configPath)) {
    await writeFile(
      configPath,
      JSON.stringify(
        {
          MODEL_PROVIDER: "",
          MODEL_NAME: "",
          MODEL_BASE_URL: "",
          MODEL_API_KEY: "",
          AGENT_WORKSPACE: "",
          LOG_LEVEL: "info"
        },
        null,
        2
      ),
      "utf8"
    );
  }

  return {
    agentHome,
    configPath
  };
}
```

> 注意：如果需要修改 `config.json`，应该用 JSON 解析和写回，不要用字符串替换。配置文件是结构化数据。

## 5. 默认工作目录

如果 `config.json` 里没有提供 `AGENT_WORKSPACE`，或者值是空字符串，就默认选择用户目录下的 `workspaceRoot`：

```ts
const defaultWorkspaceRoot = path.join(homedir(), "workspaceRoot");
```

在 Windows 上通常是：

```text
C:\Users\Administrator\workspaceRoot
```

在 macOS / Linux 上通常是：

```text
/Users/<你>/workspaceRoot
/home/<你>/workspaceRoot
```

这个目录是真正给 Agent 操作项目文件的地方。它和 `agentHome` 分开，避免把运行状态、配置、日志和用户项目文件混在一起。

如果要确保默认工作目录存在，可以新增 `createAgentWorkSpace()`：

```ts
export function getDefaultWorkspaceRoot() {
  return path.join(homedir(), "workspaceRoot");
}

export async function createAgentWorkSpace(workspaceRoot: string) {
  await mkdir(workspaceRoot, { recursive: true });
  return workspaceRoot;
}
```

## 6. 配置读取方式

以前课程里直接从 `.env` 读取：

```ts
import "dotenv/config";

export const workspaceRoot = path.resolve(env.AGENT_WORKSPACE);
```

本章之后，配置来源要调整为：

```text
1. 先确定 MINI_AGENT_HOME。
2. 调用 createHomeRoot()，确保 config.json 存在。
3. 从 config.json 读取主要配置。
4. 检查模型配置是否为空。
5. 如果模型配置为空，启动交互式输入流程，让用户补齐后写回 config.json。
6. 如果 AGENT_WORKSPACE 没有提供，则使用 path.join(homedir(), "workspaceRoot")。
7. 必要时再用环境变量覆盖敏感或临时配置。
```

推荐优先级：

```text
默认值
  < agentHome/config.json
  < 环境变量
  < 未来的命令行参数
```

也就是说：

```text
普通长期配置：写入 config.json
临时覆盖配置：使用环境变量
一次性命令配置：未来使用 CLI 参数
```

这样做的好处是：用户第一次启动后就会有一份可编辑、可迁移的本地配置文件，而不是把所有东西都塞进项目仓库里的 `.env`。

## 7. 启动时补全模型配置

模型配置不能写死为 DeepSeek，也不应该要求用户必须提前手动编辑 `config.json`。更好的体验是：程序启动时读取配置，如果发现关键字段为空，就打开交互式输入流程。

需要检查的字段：

```text
MODEL_PROVIDER   # 模型服务商名称，例如 openai、deepseek、ollama、custom
MODEL_NAME       # 具体模型名称，例如 gpt-4.1、deepseek-chat、qwen3
MODEL_BASE_URL   # 模型服务地址，优先支持 OpenAI-compatible API
MODEL_API_KEY    # API Key；本地模型如果不需要，可以允许为空
```

交互式输入可以先用命令行窗口实现，后续如果有 TUI 再升级成真正的弹窗或表单。课程当前阶段推荐使用 Node.js 的 `readline/promises`：

```ts
import { createInterface } from "node:readline/promises";
import { stdin as input, stdout as output } from "node:process";

export async function promptMissingModelConfig(config: AgentConfig) {
  const rl = createInterface({ input, output });

  try {
    if (!config.MODEL_PROVIDER) {
      config.MODEL_PROVIDER = await rl.question("模型服务商 MODEL_PROVIDER: ");
    }

    if (!config.MODEL_NAME) {
      config.MODEL_NAME = await rl.question("模型名称 MODEL_NAME: ");
    }

    if (!config.MODEL_BASE_URL) {
      config.MODEL_BASE_URL = await rl.question("模型接口地址 MODEL_BASE_URL: ");
    }

    if (!config.MODEL_API_KEY) {
      config.MODEL_API_KEY = await rl.question("模型 API Key MODEL_API_KEY: ");
    }
  } finally {
    rl.close();
  }

  return config;
}
```

补齐后要把结果写回 `agentHome/config.json`：

```ts
await writeFile(configPath, JSON.stringify(config, null, 2), "utf8");
```

这样用户第一次运行时只需要按提示输入一次，之后程序就能直接从 `config.json` 读取配置。

如果某些模型服务不需要 `MODEL_API_KEY`，可以允许用户直接回车跳过；但 `MODEL_PROVIDER`、`MODEL_NAME` 和 `MODEL_BASE_URL` 应该尽量要求填写，否则程序不知道该连接哪个模型。

## 8. 工作目录路径模块

创建或更新 `src/workspace/path.ts`：

```ts
import path from "node:path";
import { homedir } from "node:os";
import { env } from "../config/index.js";

export function getDefaultWorkspaceRoot() {
  return path.join(homedir(), "workspaceRoot");
}

export const workspaceRoot = path.resolve(
  env.AGENT_WORKSPACE || getDefaultWorkspaceRoot()
);

export function resolveWorkspacePath(inputPath: string) {
  const resolvedPath = path.resolve(workspaceRoot, inputPath);
  const relativePath = path.relative(workspaceRoot, resolvedPath);

  if (relativePath.startsWith("..") || path.isAbsolute(relativePath)) {
    throw new Error(`Path is outside workspace: ${inputPath}`);
  }

  return resolvedPath;
}

export function toWorkspaceRelativePath(fullPath: string) {
  return path.relative(workspaceRoot, fullPath) || ".";
}
```

这个模块做三件事：

| 函数 | 作用 |
| --- | --- |
| `workspaceRoot` | 把配置里的 `AGENT_WORKSPACE` 转成绝对路径；没有配置时使用用户目录下的 `workspaceRoot` |
| `resolveWorkspacePath()` | 把用户输入路径转成安全的绝对路径，并阻止越界 |
| `toWorkspaceRelativePath()` | 把绝对路径转回相对路径，方便输出 |

这里使用 `path.relative()` 判断是否越界，而不是简单使用 `startsWith()`。原因是路径前缀可能误判，例如：

```text
E:\workspace\app
E:\workspace\application
```

第二个路径虽然以相似字符串开头，但并不在第一个目录里。

## 9. 输入路径如何被解析

假设最终得到的 `workspaceRoot` 是：

```text
C:\Users\Administrator\workspaceRoot
```

那么：

```ts
resolveWorkspacePath("src/main.ts");
```

会得到：

```text
C:\Users\Administrator\workspaceRoot\src\main.ts
```

如果输入：

```ts
resolveWorkspacePath("../.env");
```

解析后会跑到工作目录外部，因此应该抛错。

第 04 章所有文件工具都要先调用 `resolveWorkspacePath()`，再读目录、读文件或搜索文本。

## 10. .env 的新定位

`.env.example` 以后可以保留少量启动覆盖项：

```env
MODEL_API_KEY=
MINI_AGENT_HOME=
LOG_LEVEL=
```

其中：

```text
MODEL_API_KEY    可以继续放在 .env 或系统环境变量里，因为它是敏感信息。
MINI_AGENT_HOME  用来改变 config.json 所在目录。
LOG_LEVEL        可以作为临时调试覆盖项。
```

不建议继续把 `AGENT_WORKSPACE=.` 作为主要默认值。默认工作目录应该由代码统一生成：

```text
path.join(homedir(), "workspaceRoot")
```

如果用户要长期修改工作目录，推荐改：

```text
agentHome/config.json
```

例如：

```json
{
  "MODEL_PROVIDER": "custom",
  "MODEL_NAME": "my-model",
  "MODEL_BASE_URL": "https://api.example.com/v1",
  "AGENT_WORKSPACE": "E:\\workspace\\some-project"
}
```

## 11. 本章暂不做什么

本章只定义本地目录、配置文件和工作目录边界，不做：

- `read_file`：第 04 章实现。
- `search_text`：第 04 章实现。

## 12. 验收思路

本章可以先通过阅读代码确认：

```text

```

重点检查：

```text
1. 没有 MINI_AGENT_HOME 时，agentHome 是否为 homedir() + ".mini-agent"。
2. createHomeRoot() 是否会创建 agentHome/config.json。
3. config.json 不存在时，是否会写入默认配置。
4. MODEL_PROVIDER / MODEL_NAME / MODEL_BASE_URL 为空时，启动时是否提示用户输入。
5. 补齐后的模型配置是否会写回 config.json。
6. AGENT_WORKSPACE 为空时，workspaceRoot 是否为 homedir() + "workspaceRoot"。
7. resolveWorkspacePath("../xxx") 是否会抛错。
```

第 04 章实现文件工具时，可以用下面方式间接验收：

```bash
npm run dev -- run "列出当前项目文件"
```

如果工具只能访问 `workspaceRoot` 内部，说明这章的目录边界已经生效。

## 13. 企业级思考

从这一章开始，项目有了两个清晰边界：

- 状态边界：配置、历史、会话、日志和缓存进入 `agentHome`。
- 文件边界：工具只能访问 `workspaceRoot`。

把配置迁移到 `agentHome/config.json` 后，项目仓库不再承担用户本地状态的职责。后续权限、审计、审批和插件系统，都会建立在这两个边界之上。
