# 03b. 工作目录与本地状态目录

## 本章目标

第 04 章会实现本地文件工具，例如：

```text
list_files
read_file
search_text
```

这些工具一旦交给 Agent 使用，就必须先有一个清晰边界：Agent 到底可以读写哪里？

所以本章只做一件事：

```text
把「Agent 可以操作的项目目录」和「CLI 自己保存状态的目录」分清楚，并写成可运行代码。
```

本章结束后，会得到两个目录：

```text
workspaceRoot  # Agent 可以访问的工作目录
agentHome      # mini-agent 自己保存配置、日志、会话的目录
```

它们的默认位置是：

```text
workspaceRoot = ~/workSpaceRoot
agentHome     = ~/.mini-agent
```

并且会实现两个初始化函数：

```ts
await createHomeRoot();
await createAgentWorkSpace();
```

最终效果：

```text
~/.mini-agent/
  config.json
  logs/
  sessions/

~/workSpaceRoot/
```

## 1. 为什么必须先做目录边界

文件工具不能直接让模型访问整台电脑。

如果没有边界，Agent 可能读到这些地方：

```text
~/.ssh
~/.codex
其他项目/.env
系统目录
```

所以从第 04 章开始，所有文件工具都应该只允许访问：

```text
workspaceRoot
```

而不能访问：

```text
agentHome
```

因为 `agentHome` 里面将来会保存配置、历史、日志、会话和缓存。

## 2. 两个目录的职责

本项目需要两个目录，不要混用。

| 目录 | 作用 | 默认路径 |
| --- | --- | --- |
| `workspaceRoot` | Agent 读写用户项目文件的地方 | `~/workSpaceRoot` |
| `agentHome` | mini-agent 自己保存状态的地方 | `~/.mini-agent` |

可以这样理解：

```text
workspaceRoot
= 给 Agent 工作的目录
= 第 04 章文件工具的安全边界
= 用户可以放项目文件、资料、文档

agentHome
= mini-agent 自己的家目录
= 保存 config.json、logs、sessions
= 不应该被 Agent 文件工具随便读取
```

## 3. 本章最终要写出的文件

本章主要修改一个文件：

```text
src/workspace/path.ts
```

最终这个文件会包含这些函数：

```ts
getAgentHome()
getDefaultWorkspaceRoot()
createHomeRoot()
readAgentConfig()
isValidWorkspacePath()
resolveWorkspaceRoot()
createAgentWorkSpace()
```

它们的关系是：

```text
createHomeRoot()
  ├─ 计算 ~/.mini-agent
  ├─ 创建 ~/.mini-agent
  ├─ 创建 logs 和 sessions
  └─ 如果 config.json 不存在，就写默认配置

createAgentWorkSpace()
  ├─ 读取 ~/.mini-agent/config.json
  ├─ 读取 AGENT_WORKSPACE
  ├─ 如果 AGENT_WORKSPACE 为空，使用 ~/workSpaceRoot
  ├─ 如果 AGENT_WORKSPACE 合法，使用配置路径
  ├─ 如果 AGENT_WORKSPACE 不合法，回退到 ~/workSpaceRoot
  └─ 创建最终工作目录
```

## 4. 先定义默认路径

打开：

```text
src/workspace/path.ts
```

先写两个最基础的路径函数：

```ts
import path from "node:path";
import { homedir } from "node:os";

export function getAgentHome() {
  return path.join(homedir(), ".mini-agent");
}

export function getDefaultWorkspaceRoot() {
  return path.join(homedir(), "workSpaceRoot");
}
```

这里不要先考虑 `.env`，也不要先考虑命令行参数。

本章先把默认规则写清楚：

```text
agentHome 默认是 ~/.mini-agent
workspaceRoot 默认是 ~/workSpaceRoot
```

## 5. 定义 config.json 的结构

`agentHome/config.json` 是 mini-agent 后续的本地配置文件。

默认内容设计为：

```json
{
  "MODEL_PROVIDER": "",
  "MODEL_NAME": "",
  "MODEL_BASE_URL": "",
  "MODEL_API_KEY": "",
  "AGENT_WORKSPACE": "",
  "LOG_LEVEL": "info"
}
```

注意这里的 `AGENT_WORKSPACE` 默认是空字符串。

它的含义是：

```text
用户还没有指定工作目录，所以程序使用默认目录 ~/workSpaceRoot。
```

不要把默认值写成 `.`。

如果写成 `.`，程序就会把当前启动目录当成 Agent 工作目录。这样不同地方启动 CLI，工作目录会变来变去，不利于教学，也不利于安全边界。

## 6. 实现 createHomeRoot

`createHomeRoot()` 只负责初始化 mini-agent 自己的状态目录。

它不创建 Agent 工作目录。

职责如下：

```text
1. 创建 ~/.mini-agent
2. 创建 ~/.mini-agent/logs
3. 创建 ~/.mini-agent/sessions
4. 如果 ~/.mini-agent/config.json 不存在，就写入默认配置
5. 返回 agentHome 和 configPath
```

代码如下：

```ts
import { existsSync } from "node:fs";
import { mkdir, writeFile } from "node:fs/promises";
import path from "node:path";
import { homedir } from "node:os";

export function getAgentHome() {
  return path.join(homedir(), ".mini-agent");
}

export function getDefaultWorkspaceRoot() {
  return path.join(homedir(), "workSpaceRoot");
}

export async function createHomeRoot() {
  const agentHome = getAgentHome();
  const configPath = path.join(agentHome, "config.json");

  await mkdir(agentHome, { recursive: true });
  await mkdir(path.join(agentHome, "logs"), { recursive: true });
  await mkdir(path.join(agentHome, "sessions"), { recursive: true });

  if (!existsSync(configPath)) {
    await writeFile(
      configPath,
      JSON.stringify(
        {
          MODEL_PROVIDER: "",
          MODEL_NAME: "",
          MODEL_BASE_URL: "",
          MODEL_API_KEY: "",
          AGENT_WORKSPACE: "",
          LOG_LEVEL: "info"
        },
        null,
        2
      ),
      "utf-8"
    );
  }

  return {
    agentHome,
    configPath
  };
}
```

这一段代码是闭环的：

```text
调用 createHomeRoot()
  -> 一定能得到 ~/.mini-agent
  -> 一定能得到 ~/.mini-agent/config.json
```

## 7. 读取 config.json

下一步要实现 `createAgentWorkSpace()`。

但它不能直接写死 `~/workSpaceRoot`，因为用户可能已经在 `config.json` 里配置了：

```json
{
  "AGENT_WORKSPACE": "/Users/you/projects/demo"
}
```

所以先写一个读取配置的函数。

继续在 `src/workspace/path.ts` 中添加：

```ts
import { readFile } from "node:fs/promises";
```

然后添加类型和函数：

```ts
type AgentConfig = {
  MODEL_PROVIDER?: string;
  MODEL_NAME?: string;
  MODEL_BASE_URL?: string;
  MODEL_API_KEY?: string;
  AGENT_WORKSPACE?: string;
  LOG_LEVEL?: string;
};

export async function readAgentConfig() {
  const { configPath } = await createHomeRoot();
  const configText = await readFile(configPath, "utf-8");

  return JSON.parse(configText) as AgentConfig;
}
```

这里故意让 `readAgentConfig()` 先调用 `createHomeRoot()`。

这样可以保证：

```text
读取 config.json 之前，config.json 一定已经存在。
```

## 8. 校验 AGENT_WORKSPACE 是否合理

用户可能把 `AGENT_WORKSPACE` 写错，例如：

```json
{
  "AGENT_WORKSPACE": ""
}
```

或者写成包含非法字符的路径。

所以我们先定义一个简单校验函数。

```ts
export function isValidWorkspacePath(workspacePath: string) {
  const normalizedPath = workspacePath.trim();

  if (!normalizedPath) {
    return false;
  }

  if (normalizedPath.includes("\0")) {
    return false;
  }

  return path.isAbsolute(normalizedPath);
}
```

这里先采用一个清晰规则：

```text
AGENT_WORKSPACE 必须是绝对路径。
```

例如这些是合理的：

```text
/Users/you/work/demo
/home/you/work/demo
D:\work\demo
```

这些是不合理的：

```text
""
.
./demo
../demo
```

为什么不建议用相对路径？

因为相对路径依赖启动目录。你在不同目录执行 `chatx`，同一个 `./demo` 会指向不同地方。

## 9. 计算最终 workspaceRoot

现在我们可以写一个函数，专门决定最终工作目录。

规则如下：

```text
1. 读取 config.json
2. 如果 AGENT_WORKSPACE 为空，返回 ~/workSpaceRoot
3. 如果 AGENT_WORKSPACE 是合理绝对路径，返回它
4. 如果 AGENT_WORKSPACE 不合理，回退到 ~/workSpaceRoot
```

代码如下：

```ts
export async function resolveWorkspaceRoot() {
  const config = await readAgentConfig();
  const configuredWorkspace = config.AGENT_WORKSPACE?.trim() ?? "";

  if (!configuredWorkspace) {
    return getDefaultWorkspaceRoot();
  }

  if (!isValidWorkspacePath(configuredWorkspace)) {
    return getDefaultWorkspaceRoot();
  }

  return configuredWorkspace;
}
```

这一步不要创建目录，只负责计算路径。

这样职责更清楚：

```text
resolveWorkspaceRoot()      只决定路径
createAgentWorkSpace()      创建目录
```

## 10. 实现 createAgentWorkSpace

最后实现本章的第二个初始化函数。

```ts
export async function createAgentWorkSpace() {
  const workspaceRoot = await resolveWorkspaceRoot();

  await mkdir(workspaceRoot, { recursive: true });

  return workspaceRoot;
}
```

它的行为是：

```text
如果 config.json 里的 AGENT_WORKSPACE 为空：
  创建 ~/workSpaceRoot

如果 config.json 里的 AGENT_WORKSPACE 是合理绝对路径：
  创建 AGENT_WORKSPACE 对应目录

如果 config.json 里的 AGENT_WORKSPACE 不合理：
  创建 ~/workSpaceRoot
```

## 11. path.ts 完整代码

本章完成后，`src/workspace/path.ts` 可以整理成下面这样。

```ts
import { existsSync } from "node:fs";
import { mkdir, readFile, writeFile } from "node:fs/promises";
import { homedir } from "node:os";
import path from "node:path";

type AgentConfig = {
  MODEL_PROVIDER?: string;
  MODEL_NAME?: string;
  MODEL_BASE_URL?: string;
  MODEL_API_KEY?: string;
  AGENT_WORKSPACE?: string;
  LOG_LEVEL?: string;
};

export function getAgentHome() {
  return path.join(homedir(), ".mini-agent");
}

export function getDefaultWorkspaceRoot() {
  return path.join(homedir(), "workSpaceRoot");
}

export async function createHomeRoot() {
  const agentHome = getAgentHome();
  const configPath = path.join(agentHome, "config.json");

  await mkdir(agentHome, { recursive: true });
  await mkdir(path.join(agentHome, "logs"), { recursive: true });
  await mkdir(path.join(agentHome, "sessions"), { recursive: true });

  if (!existsSync(configPath)) {
    await writeFile(
      configPath,
      JSON.stringify(
        {
          MODEL_PROVIDER: "",
          MODEL_NAME: "",
          MODEL_BASE_URL: "",
          MODEL_API_KEY: "",
          AGENT_WORKSPACE: "",
          LOG_LEVEL: "info"
        },
        null,
        2
      ),
      "utf-8"
    );
  }

  return {
    agentHome,
    configPath
  };
}

export async function readAgentConfig() {
  const { configPath } = await createHomeRoot();
  const configText = await readFile(configPath, "utf-8");

  return JSON.parse(configText) as AgentConfig;
}

export function isValidWorkspacePath(workspacePath: string) {

  if (!normalizedPath) {
    return false;
  }

  if (normalizedPath.includes("\0")) {
    return false;
  }

  return path.isAbsolute(normalizedPath);
}

export async function resolveWorkspaceRoot() {
  const config = await readAgentConfig();

```

## 16. 本章和 .env 的关系

前面章节用 `.env` 保存 DeepSeek 配置，这是为了快速跑通模型调用。

但从工程化角度看，长期配置更适合放在：

```text
~/.mini-agent/config.json
```

本章先把 `config.json` 建起来，但暂时不要求你立刻把模型配置全部迁过去。

当前阶段可以这样分工：

```text
.env
  临时保存 DEEPSEEK_API_KEY、DEEPSEEK_MODEL、DEEPSEEK_BASE_URL

~/.mini-agent/config.json
  保存 CLI 自己的长期配置，例如 AGENT_WORKSPACE、LOG_LEVEL
```

后续可以继续演进成：

```text
模型配置也从 config.json 读取
.env 只作为环境变量覆盖项
```

不要在本章同时改太多东西，否则教程会变乱。

## 18. 本章小结

本章只解决一个基础问题：

```text
Agent 到底可以操作哪个目录？mini-agent 自己的状态又保存在哪里？
```

最终规则是：

```text
1. createHomeRoot() 初始化 ~/.mini-agent。
2. config.json 保存在 ~/.mini-agent/config.json。
3. AGENT_WORKSPACE 为空时，使用 ~/workSpaceRoot。
4. AGENT_WORKSPACE 是合理绝对路径时，使用配置路径。
5. AGENT_WORKSPACE 不合理时，回退到 ~/workSpaceRoot。
6. createAgentWorkSpace() 负责创建最终 workspaceRoot。
```

这样第 04 章的文件工具就有了明确、安全、可解释的根目录。
