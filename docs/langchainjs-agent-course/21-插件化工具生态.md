# 21. 插件化工具生态：让 Agent 可扩展

## 本章目标

第 20 章定义了权限边界。本章在这个边界内开放插件机制，让业务工具可以持续接入，而不污染核心运行时。

随着工具变多，不能把所有工具都写死在 `src/tools/index.ts`。本章设计插件化工具系统。

目标：

```text
plugins/
  github/
  jira/
  slack/
  database/
```

## 1. 为什么需要插件化

企业 Agent 会不断接入：

- GitHub / GitLab。
- Jira / Linear。
- Slack / 飞书。
- 数据库。
- CI/CD。
- 监控平台。

如果都写进核心项目，维护会失控。

## 2. 插件接口

创建 `src/plugins/types.ts`：

```ts
import type { StructuredToolInterface } from "@langchain/core/tools";

export type ToolPlugin = {
  name: string;
  description: string;
  tools: StructuredToolInterface[];
  permissions: string[];
};
```

## 3. 插件注册器

```ts
const plugins: ToolPlugin[] = [];

export function registerPlugin(plugin: ToolPlugin) {
  plugins.push(plugin);
}

export function getPluginTools() {
  return plugins.flatMap((plugin) => plugin.tools);
}
```

## 4. 插件配置

`.env` 或配置文件：

```json
{
  "plugins": {
    "github": { "enabled": true },
    "jira": { "enabled": false }
  }
}
```

启动时只加载启用的插件。

## 5. 插件安全

插件必须声明：

- 需要哪些权限。
- 是否会访问网络。
- 是否会写入外部系统。
- 是否需要审批。
- 输出是否可能包含敏感数据。

## 6. 示例插件：GitHub 只读

工具：

```text

github_search_code
```

先做只读，再考虑写操作。

## 7. 验收


```text

- list_files

- github_list_issues
```

## 8. 企业级思考

插件化能让 Agent 平台化。核心项目只负责运行时、权限、观测和协议；业务团队用插件扩展能力。
