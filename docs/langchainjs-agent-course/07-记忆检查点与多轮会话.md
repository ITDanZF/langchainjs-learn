# 07. 会话记忆：从 messages 到 thread

## 本章目标

当前 `mini-agent-langchain` 已经可以进入 CLI，并把用户输入交给模型流式输出。但它还没有真正的“记忆”。

现在的调用链是：

```text
main.ts
  -> Bootstrap.setup()
  -> CLI.run()
  -> AgentRuntime.model.stream(input)
  -> PrintStream(result)
```

这意味着每一轮输入都是单独发送给模型的。用户输入：

```text
> 我叫张三
> 我刚才说我叫什么？
```

如果不把第一轮消息放进第二轮上下文，模型就不能稳定回答。

本章要解决三个问题：

1. 什么是短期记忆。
2. 为什么要有 `threadId`。
3. 当前项目如何先实现最小 Memory，再理解 LangGraph 官方 checkpointer。

## 1. 短期记忆是什么

短期记忆就是当前对话线程里的消息历史：

```text
thread_id = default

user: 我叫张三
assistant: 好的，我记住了
user: 我刚才说我叫什么？
```

它解决的问题是：

```text
同一个会话里，Agent 如何知道前面说过什么？
```

短期记忆通常保存：

- 用户消息
- AI 回复
- 工具调用结果
- 当前任务状态
- 对话摘要

当前阶段先只做最简单的 `messages[]`。

## 2. 为什么需要 threadId

如果只有一份全局 messages，所有对话都会混在一起。

更合理的结构是：

```text
thread: default  -> 默认对话上下文
thread: docs     -> 文档学习上下文
thread: debug    -> 调试任务上下文
```

同一个 `threadId` 复用同一份消息历史，不同 `threadId` 互相隔离。

用数据结构表示就是：

```ts
Map<string, BaseMessageLike[]>
```

也就是：

```text
threadId -> messages
```

## 3. 当前项目应该先手写 Memory

当前项目还没有 LangGraph graph，所以不要急着直接套 `MemorySaver`。更好的学习顺序是先在 `src/Memory/index.ts` 中实现一个最小短期记忆管理器。

建议接口：

```ts
import type { BaseMessageLike } from "@langchain/core/messages";

export type ThreadId = string;

export default class Memory {
  private threads = new Map<ThreadId, BaseMessageLike[]>();

  getMessages(threadId: ThreadId) {
    return this.threads.get(threadId) ?? [];
  }

  addMessage(threadId: ThreadId, message: BaseMessageLike) {
    const messages = this.getMessages(threadId);
    this.threads.set(threadId, [...messages, message]);
  }

  clear(threadId: ThreadId) {
    this.threads.delete(threadId);
  }
}
```

这一版只保存在进程内，程序退出就丢失。它不是最终方案，但非常适合理解记忆的本质。

## 4. Model 层如何接收历史消息

当前 `Model.stream(input)` 接收的是单条输入。要支持短期记忆，需要让模型调用可以接收完整 messages。

概念上，模型调用应该从：

```ts
stream(input: BaseMessageLike) {
  return this.getChain().stream({ input });
}
```

逐步演进为：

```ts
streamMessages(messages: BaseMessageLike[]) {
  return this.CurrentModel.stream(messages);
}
```

或者让 prompt 模板支持 `MessagesPlaceholder`，把历史消息插入 prompt。

本课程在这一章先强调概念边界：**记忆模块负责保存 messages，模型模块负责消费 messages。**

## 5. CLI 如何使用 Memory

理想的主流程是：

```text
用户输入
  -> memory.addMessage(threadId, human message)
  -> model.streamMessages(memory.getMessages(threadId))
  -> 收集 assistant 回复
  -> memory.addMessage(threadId, assistant message)
```

也就是：

```ts
const threadId = "default";

memory.addMessage(threadId, { role: "user", content: input });

const result = await model.streamMessages(memory.getMessages(threadId));
const assistantText = await collectStream(result);

memory.addMessage(threadId, { role: "assistant", content: assistantText });
```

注意：当前项目的 `PrintStream` 只负责打印，不负责收集完整文本。如果要把 AI 回复写入记忆，需要新增一个“边打印边收集”的工具函数。

## 6. 命令设计

短期记忆跑通后，CLI 可以支持这些命令：

```text
/exit              退出
/clear             清空当前 thread
/thread <id>       切换 thread
/threads           查看当前进程里的 thread
```

其中最核心的是：

```text
/thread docs
```

它让你可以在不同任务之间切换上下文。

## 7. session 落盘

进程内 Memory 的缺点是程序退出后会丢失。当前项目已经创建了：

```text
~/.mini-agent/sessions
```

因此下一步可以把每条消息追加到：

```text
~/.mini-agent/sessions/<thread-id>.jsonl
```

示例：

```json
{"role":"user","content":"我叫张三","createdAt":"2026-06-30T00:00:00.000Z"}
{"role":"assistant","content":"好的，我记住了。","createdAt":"2026-06-30T00:00:01.000Z"}
```

JSONL 的好处是：

- 追加写入简单。
- 可以按行恢复。
- 方便调试和审计。
- 不需要数据库。

## 8. LangGraph 官方做法

LangGraph 官方把短期记忆称为 thread-level persistence。核心 API 是：

```ts
import { MemorySaver, StateGraph } from "@langchain/langgraph";

const checkpointer = new MemorySaver();
const graph = builder.compile({ checkpointer });
```

调用时传入：

```ts
await graph.invoke(
  { messages: [{ role: "user", content: "hi! i am Bob" }] },
  { configurable: { thread_id: "1" } },
);
```

这里的关键是：

```text
checkpointer 保存图状态
thread_id 区分对话线程
messages 是 state 的一部分
```

也就是说，LangGraph 的短期记忆不是一个单独的聊天数组，而是整个 graph state 的持久化。

## 9. 当前手写 Memory 与 LangGraph checkpointer 的关系

| 当前手写版本 | LangGraph 官方版本 |
| --- | --- |
| `Memory` class | `checkpointer` |
| `Map<threadId, messages>` | thread-level state |
| `getMessages(threadId)` | `graph.getState(config)` |
| `clear(threadId)` | `checkpointer.deleteThread(threadId)` |
| 手动拼 messages | graph 自动合并 state |

所以手写 Memory 不是弯路。它帮助你理解 LangGraph 之后替你管理了什么。

## 10. 长期记忆先不要做

长期记忆是跨 thread、跨 session 的信息，例如：

```text
用户偏好中文回答
项目配置来自 config.json
用户希望先讲概念再写代码
```

LangGraph 官方使用 `store` 处理长期记忆：

```ts
import { InMemoryStore } from "@langchain/langgraph";

const store = new InMemoryStore();
const graph = builder.compile({ store });
```

但当前项目还不应该马上做长期记忆。原因是长期记忆需要回答：

- 什么信息值得保存？
- 谁允许保存？
- 如何删除？
- 如何避免保存敏感信息？
- 如何检索相关记忆？

当前阶段先把短期记忆跑通。

## 11. 本章验收

完成本章后，至少应该能做到：

```text
> 我叫张三
> 我刚才说我叫什么？
```

Agent 能在同一个 thread 中回答：

```text
你刚才说你叫张三。
```

然后切换 thread：

```text
> /thread fresh
> 我叫什么？
```

Agent 不应该知道“张三”，因为这是另一个 thread。
