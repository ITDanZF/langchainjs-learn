# 02. 运行链路

## 1. 一次典型代码任务的链路

```text
用户请求
  -> 读取当前会话和系统/开发者指令
  -> 定位工作区和项目配置
  -> 读取 AGENTS.md / 项目文档 / 相关文件
  -> 发现适用 skills / plugins / MCP tools
  -> 模型生成分析或计划
  -> 如需工具：发起工具调用
  -> 沙箱和审批层判断是否允许
  -> 工具执行并返回结果
  -> 模型基于结果继续推理
  -> 生成 patch / 命令 / 报告 / 进一步问题
  -> 将结果写回会话、日志和状态
```

## 2. 工具调用不是“直接执行”

模型不能任意访问文件系统或操作系统。它只能请求运行时暴露的工具，运行时再根据：

- 工具定义；
- 当前 sandbox mode；
- writable roots；
- network policy；
- approval policy；
- 平台安全限制；
- 用户确认；

决定是否执行。

这也是为什么同一个命令在普通终端可执行，但在 Codex 内可能失败或要求审批。

## 3. 规划和执行是交替的

Codex 的 agent 行为不是一次性生成完整答案，而是一个 Observe-Think-Act 循环：

```text
观察：读取文件、搜索、查看配置、运行测试
思考：判断影响范围、选择方案、识别风险
行动：修改文件、运行命令、打开浏览器、生成报告
校验：读取结果、修复问题、总结残余风险
```

工具结果会进入后续上下文，影响下一步推理。

## 4. 上下文选择策略

Codex 不会把整个仓库、所有历史、所有 skill 全部加载进模型。更常见的策略是：

- 先加载高优先级指令和任务消息。
- 根据工作目录加载相关 `AGENTS.md`。
- 根据用户请求和 skill description 选择 skill。
- 只读取相关源码文件、配置和文档。
- 对历史上下文进行压缩或摘要。
- 对长资源采用按需读取。

这是一种上下文预算管理策略。它降低 token 成本，也减少无关信息干扰。

## 5. 审批分叉

执行链路中常见分叉：

```text
工具请求
  -> 只读且允许：直接执行
  -> 写入工作区内且允许：直接执行
  -> 写入工作区外：需要审批或失败
  -> GUI / 外部应用：需要审批或失败
  -> 网络受限：需要审批或失败
  -> 潜在破坏性命令：需要明确审批
```

本机实际表现中，Windows sandbox helper 缺失会导致普通 sandbox 执行失败，提升权限后才能读取或写入部分本地文件。这说明执行层依赖平台侧 helper 程序和运行时集成。

## 6. 结果回写

工具执行后的结果会以几种形式留存：

- 当前对话上下文中的工具返回值。
- 本地日志数据库。
- 会话索引和历史文件。
- 生成或修改的项目文件。
- 可选 memories。
- 插件和浏览器状态缓存。

这让 Codex 可以在同一线程内保持连续性，并在部分场景下跨线程保留偏好或事实。
