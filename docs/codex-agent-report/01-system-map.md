# 01. Codex Agent 系统总览

## 1. 分层视角

从工程角度看，Codex agent 系统可以拆成七层：

```text
┌──────────────────────────────────────────────┐
│ 1. 用户交互层：CLI / IDE / Desktop / Cloud    │
├──────────────────────────────────────────────┤
│ 2. 会话编排层：thread、history、compaction     │
├──────────────────────────────────────────────┤
│ 3. 指令层：system/developer/user/AGENTS/config │
├──────────────────────────────────────────────┤
│ 4. 能力发现层：tools、skills、plugins、MCP      │
├──────────────────────────────────────────────┤
│ 5. 模型规划层：分析、计划、选择工具、生成补丁   │
├──────────────────────────────────────────────┤
│ 6. 安全执行层：sandbox、approval、权限策略      │
├──────────────────────────────────────────────┤
│ 7. 状态持久层：logs、sessions、memories、cache  │
└──────────────────────────────────────────────┘
```

这些层不是严格的进程边界，而是职责边界。Codex 的核心价值在于把它们组合成一个可持续工作的 coding agent。

## 2. 关键对象

### Thread / Session

线程承载用户对话、工具调用结果、压缩摘要和当前任务状态。上下文窗口不足时，系统会保留关键状态并压缩旧上下文。

### Instruction Surfaces

Codex 接收多类指令：

- 系统和开发者指令：由运行环境注入，优先级最高。
- 用户消息：当前任务目标和约束。
- `AGENTS.md`：仓库或目录级长期工程约定。
- `config.toml`：模型、沙箱、MCP、插件、项目可信配置。
- Skill 文档：可复用任务流程。
- Plugin manifest：可安装能力包的元信息和入口。

### Capability Registry

Codex 的能力不是硬编码在单个 prompt 中，而是由多种注册面汇聚：

- 内置工具，例如 shell、patch、web、browser、node_repl。
- MCP server 暴露的工具或资源。
- Skills 暴露的工作流和资源。
- Plugins 打包的 skills、MCP、apps、assets、hooks。
- Subagents 暴露的委托能力。

### Execution Sandbox

所有工具执行都受沙箱和审批策略约束。写文件、网络、GUI、危险命令、越界路径都会触发不同级别的限制或确认。

## 3. 运行系统的基本性质

Codex agent runtime 有几个明显特征：

- **上下文驱动**：模型每轮推理依赖当前注入的上下文，而不是永久知道所有历史。
- **渐进加载**：skills、references、plugin 资源不会全部塞进上下文；先加载元数据，再按触发读取正文和资源。
- **工具闭环**：模型通过工具观察外部状态，再把结果并入下一轮推理。
- **安全优先**：工具执行前由沙箱、路径白名单、审批策略、权限配置共同约束。
- **本地与远程混合**：本地 CLI/Desktop 负责文件、进程和插件状态；模型推理和部分产品状态由 OpenAI 服务端参与。

## 4. 与传统 IDE agent 的差异

Codex 不是只在编辑器内工作的 autocomplete agent。它更像一个跨表面的 agent runtime：

- CLI 更偏本地仓库和终端工作流。
- IDE extension 更贴近编辑器上下文。
- Desktop app 更偏计划、审查、交互式工作。
- Cloud/Web 更适合托管并行任务。
- Browser/Chrome 插件扩展到网页和浏览器状态。

这解释了为什么 Codex 的工程系统需要插件、MCP、沙箱、状态数据库和多种本地目录，而不是只依赖一个 prompt 文件。
