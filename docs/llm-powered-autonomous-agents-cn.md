# LLM 驱动的自主智能体

> 原文：Lilian Weng, [LLM Powered Autonomous Agents](https://lilianweng.github.io/posts/2023-06-23-agent/), 2023-06-23  
> 本文档是中文整理版，偏技术笔记与结构化解读，不是逐句翻译。

## 1. 文章要解决的问题

大语言模型不仅可以生成文本，也可以被组织成一个“自主智能体”的核心控制器。文章讨论的问题是：如果把 LLM 当作智能体的大脑，它还需要哪些外部组件，才能完成复杂、长期、需要与环境交互的任务？

原文给出的基本答案是：一个 LLM 驱动的智能体通常由三类关键能力组成。

1. **规划（Planning）**：把大任务拆成小任务，形成行动步骤，并在执行过程中反思和修正。
2. **记忆（Memory）**：保存短期上下文与长期经验，在需要时检索相关信息。
3. **工具使用（Tool Use）**：调用外部 API、搜索引擎、代码解释器、数据库、计算器等能力，弥补模型自身知识和行动能力的限制。

可以把整体结构理解为：

```text
用户目标
  ↓
LLM 作为控制中心
  ├─ 规划：任务拆解、反思、重规划
  ├─ 记忆：短期上下文、长期向量检索
  └─ 工具：搜索、计算、代码、API、外部环境
  ↓
行动、观察、反馈、继续迭代
```

## 2. Agent 的核心架构

### 2.1 LLM 是“大脑”

在这类系统中，LLM 负责理解目标、生成计划、选择工具、解释观察结果，并决定下一步行动。它类似一个通用推理与控制模块。

但单独的 LLM 有明显限制：

- 上下文窗口有限，无法天然保存长期状态。
- 参数知识可能过时，无法保证实时性。
- 复杂任务需要多步执行，单次生成很容易出错。
- 只能生成文本，不能直接访问外部世界。

因此，文章将 LLM 放进一个更大的 agent 系统中，通过规划、记忆和工具来补齐这些短板。

### 2.2 规划：从目标到行动序列

规划模块让 agent 能够把抽象目标拆解成可执行步骤，并在每一步后根据反馈调整方向。文章主要讨论两类能力：任务分解与自我反思。

#### 任务分解

任务分解是把复杂目标拆成多个较小、较清晰的子任务。常见方法包括：

- **Chain of Thought（CoT）**：让模型显式写出中间推理步骤，提升复杂推理表现。
- **Tree of Thoughts（ToT）**：不只生成一条推理链，而是探索多条候选路径，并通过搜索或打分选择更好的路径。
- **任务专用提示词**：例如要求模型“先写计划，再逐步执行”。
- **人类输入**：在关键节点由人提供任务拆解或纠偏。

CoT 更像线性推理，ToT 则把问题空间看成树状结构，允许模型回溯、比较和选择。

#### 自我反思

复杂任务中，agent 不可能一次生成完美方案。自我反思机制让它能基于历史行动和环境反馈改进后续步骤。

文章提到的典型方法包括：

- **ReAct**：把推理（reasoning）和行动（acting）交替结合。模型先思考，再调用工具或执行动作，然后观察结果，再继续推理。
- **Reflexion**：把失败经验转化为语言形式的记忆，让模型在后续尝试中避免重复错误。
- **Chain of Hindsight（CoH）**：利用成功和失败轨迹进行学习，让模型从反馈中总结更好的行为模式。
- **Algorithm Distillation（AD）**：把强化学习式的历史轨迹放进上下文，让模型学习产生更优策略。

自我反思的价值在于：它把一次性回答变成了迭代式求解，让 agent 更接近“试错、观察、修正”的工作方式。

## 3. 记忆系统

文章把记忆分成几类，并将认知科学中的记忆概念映射到 LLM agent。

### 3.1 人类记忆的类比

常见划分包括：

- **感觉记忆（Sensory Memory）**：短暂保存视觉、听觉等感官输入。
- **短期记忆 / 工作记忆（Short-Term / Working Memory）**：保存当前正在处理的信息。
- **长期记忆（Long-Term Memory）**：长期保存知识、经验和事件。

映射到 LLM agent 中：

- LLM 的上下文窗口类似短期记忆。
- 向量数据库、外部文档库、历史日志等类似长期记忆。
- 多模态输入可以类比为感觉记忆的一部分。

### 3.2 短期记忆

短期记忆主要由 prompt 和上下文窗口承载。它能保存当前对话、任务状态、最近工具调用结果等信息。

限制也很明显：

- 上下文窗口有限。
- 长上下文会增加成本和延迟。
- 太多无关信息会干扰模型注意力。

因此，agent 通常需要选择性保留上下文，而不是把所有历史都塞给模型。

### 3.3 长期记忆

长期记忆通常借助外部存储实现，常见方案是：

1. 把文本、经验、观察结果切分成片段。
2. 使用 embedding 模型转成向量。
3. 存入向量数据库。
4. 在需要时根据语义相似度检索相关片段。
5. 把检索结果放回 prompt，辅助当前决策。

这类机制也被称为 **Maximum Inner Product Search（MIPS）** 或基于向量相似度的语义检索。

### 3.4 记忆检索的挑战

长期记忆不是简单“存起来就好”，还需要解决：

- 检索结果是否真的相关。
- 旧信息是否过时。
- 记忆是否需要摘要、合并或遗忘。
- 重要经验如何提高检索优先级。
- 错误观察是否会污染后续决策。

好的记忆系统需要兼顾相关性、时效性、重要性和可压缩性。

## 4. 工具使用

工具使用是 LLM agent 从“会说”走向“能做”的关键。LLM 可以通过工具访问外部能力，例如：

- 搜索引擎：获取最新信息。
- 计算器：处理精确计算。
- 代码解释器：执行程序、分析数据、验证假设。
- 数据库：查询结构化信息。
- 外部 API：订票、发邮件、操作软件系统。
- 专用模型：图像生成、语音识别、机器人控制等。

### 4.1 MRKL

MRKL（Modular Reasoning, Knowledge and Language）强调把神经模型和符号工具结合起来。LLM 负责理解和路由，外部专家模块负责专业任务。

例如：

- 数学问题交给计算器。
- 当前事件交给搜索。
- 数据查询交给数据库。
- 常识推理交给语言模型。

这种架构能减少 LLM 在精确计算和事实检索上的幻觉。

### 4.2 TALM 与 Toolformer

文章提到的研究方向包括让模型学会调用工具：

- **TALM（Tool Augmented Language Models）**：通过工具增强语言模型，让模型在生成过程中调用外部工具。
- **Toolformer**：让模型从少量示例中自监督学习何时调用 API、调用什么 API、如何使用返回结果。

核心思想是：工具调用不应完全依赖人工规则，模型可以学习在合适的上下文中主动使用工具。

### 4.3 工具使用的难点

工具使用带来能力扩展，也带来新的系统问题：

- 工具选择错误会导致任务偏离。
- 工具参数可能格式不合法。
- 工具返回结果需要被正确解释。
- 外部 API 可能失败、超时或返回脏数据。
- 多工具链路会放大中间错误。
- 安全权限需要严格控制。

所以实践中的 agent 往往需要工具 schema、参数校验、错误恢复、调用日志和权限边界。

## 5. 典型案例

### 5.1 ChemCrow

ChemCrow 是面向化学任务的 LLM agent。它把 LLM 和一组化学相关工具结合起来，用于完成分子合成、性质预测等任务。

这个案例说明：在专业领域中，LLM 不一定要自己掌握所有专业能力，而是可以调度可靠的领域工具。

### 5.2 Generative Agents

Generative Agents 研究构建了一个模拟小镇，让多个由 LLM 驱动的角色在环境中生活、交流和行动。

该系统强调三类机制：

- **记忆流**：记录 agent 的经历。
- **反思**：从具体经历中总结更高层次的认识。
- **规划**：生成日程和后续行动。

这个案例展示了记忆与反思如何让 agent 形成更连续、更像人的行为轨迹。

### 5.3 AutoGPT

AutoGPT 是较早流行的通用自主 agent 项目之一。用户给出目标后，系统会尝试自动拆解任务、调用工具、保存记忆并持续推进。

它展示了 LLM agent 的潜力，也暴露了早期通用 agent 的问题：

- 容易陷入循环。
- 计划不稳定。
- 工具调用成本高。
- 结果可靠性不足。
- 对任务边界和停止条件掌握较差。

## 6. 文章总结出的主要挑战

### 6.1 上下文长度限制

LLM 的上下文窗口限制了短期记忆容量。即使模型支持长上下文，也仍然会面临成本、延迟和注意力稀释问题。

### 6.2 长期规划困难

LLM 在短步骤推理中表现较好，但长期、多阶段任务仍然困难。问题包括：

- 初始计划容易不完整。
- 中间失败会影响后续步骤。
- 很难稳定评估“当前是否接近目标”。
- 任务越长，错误累积越明显。

### 6.3 自然语言接口的可靠性

很多 agent 使用自然语言作为模块之间的接口，例如模型输出一段文字，再由系统解析成行动。这样灵活但脆弱：

- 表达可能含糊。
- 格式可能漂移。
- 模型可能编造不存在的工具或参数。
- 输出不一定满足机器可执行约束。

因此，工程实践中常结合结构化输出、函数调用、schema 校验和状态机来提升可靠性。

## 7. 对工程实践的启发

如果要构建一个 LLM agent，可以从以下设计原则出发：

1. **先定义任务边界**：明确 agent 能做什么、不能做什么、何时停止。
2. **让计划可观察**：把任务拆解、当前状态、下一步行动记录下来。
3. **让工具强约束**：用 schema、类型校验和权限控制约束工具调用。
4. **让记忆可检索也可遗忘**：不要无限堆积历史，定期摘要、合并、淘汰。
5. **把反思变成数据**：失败原因、成功策略和用户偏好都可以进入记忆。
6. **用评估闭环约束行为**：为关键任务设计验收标准、测试和人工确认节点。
7. **避免过度自主**：在高风险操作前加入用户确认或策略限制。

## 8. 一句话总结

这篇文章的核心观点是：LLM agent 不是“一个更长的 prompt”，而是一个由 LLM、规划机制、记忆系统和外部工具共同组成的闭环系统。LLM 提供通用推理和语言控制能力，规划让它能拆解和修正任务，记忆让它能跨时间积累经验，工具让它能真正接触外部世界。
