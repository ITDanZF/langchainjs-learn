# 04. 第一次调用真实大模型

## 本章目标

上一章已经完成了最小 CLI 对话程序：

```text
命令行输入
  ↓
构造 messages
  ↓
追加一条模拟 assistant 回复
  ↓
打印消息列表
```

本章先不急着封装 `LLMClient`，而是先真实调用一次大模型，让程序得到真正的模型回复：

```text
命令行输入
  ↓
构造 messages
  ↓
把 messages 发送给大模型 API
  ↓
拿到模型返回内容
  ↓
追加为 assistant 消息
  ↓
打印消息列表
```

这一章的目标很明确：先证明 API Key、模型名、请求地址、请求格式都能跑通。下一章再把这段调用封装成可复用的客户端。

## 1. 为什么先直接调用，再封装

如果还没有成功调用过真实模型，就先设计封装层，很容易不知道自己到底在封装什么。

更适合初学的顺序是：

```text
先直接调用一次真实 API
  ↓
确认请求和响应格式
  ↓
发现重复代码和边界
  ↓
再抽成 LLMClient
```

这样封装不是凭空设计出来的，而是从已经跑通的代码里提炼出来的。

## 2. 本章需要学习的新工具：fetch

本章会用 Node.js 内置的 `fetch` 发送 HTTP 请求。

`fetch` 可以理解成 JavaScript 里发网络请求的工具：

```ts
const response = await fetch("https://example.com", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
  },
  body: JSON.stringify({ hello: "world" }),
});
```

Node.js 18+ 已经内置 `fetch`，不需要额外安装 `axios` 或 `node-fetch`。

本章只需要掌握四个点：

- `method: "POST"`：向服务器提交数据。
- `headers`：告诉服务器请求格式，并传递 API Key。
- `body`：请求体，通常要用 `JSON.stringify(...)` 转成 JSON 字符串。
- `await response.json()`：把服务器返回的 JSON 解析成对象。

## 3. 确认环境变量

第 02 章已经创建过 `.env`。这里确认里面有两个变量：

```env
DEEPSEEK_API_KEY=你的_api_key
DEEPSEEK_MODEL=deepseek-chat
```

注意变量名必须和代码一致：

```text
DEEPSEEK_API_KEY
DEEPSEEK_MODEL
```

不要写成 `DEEEPSEEK_API_KEY`，多一个 `E` 会导致代码读不到。

`src/config.ts` 保持这样：

```ts
import "dotenv/config";

export const config = {
  DEEPSEEK_API_KEY: process.env.DEEPSEEK_API_KEY || "",
  DEEPSEEK_MODEL: process.env.DEEPSEEK_MODEL || "deepseek-chat",
};
```

这里给 `DEEPSEEK_MODEL` 一个默认值，是为了避免 `.env` 只配置了 Key 时程序直接拿到空模型名。

## 4. 先检查 API Key 是否存在

在真实请求前，先让程序检查配置。

如果没有 API Key，应该提前报错，而不是发一个一定失败的请求。

在 `src/index.ts` 中导入配置：

```ts
import { config } from "./config";
```

然后在发请求前检查：

```ts
if (!config.DEEPSEEK_API_KEY) {
  console.error("缺少 DEEPSEEK_API_KEY，请先在 .env 中配置。这个值不能提交到 Git。");
  process.exit(1);
}
```

## 5. 编写一次真实请求

DeepSeek 提供 OpenAI 兼容的 Chat Completions 接口。最小请求地址是：

```text
https://api.deepseek.com/chat/completions
```

在 `src/index.ts` 中添加一个函数：

```ts
async function callDeepSeek(messages: ChatMessage[]): Promise<string> {
  const response = await fetch("https://api.deepseek.com/chat/completions", {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      Authorization: `Bearer ${config.DEEPSEEK_API_KEY}`,
    },
    body: JSON.stringify({
      model: config.DEEPSEEK_MODEL,
      messages,
    }),
  });

  if (!response.ok) {
    const errorText = await response.text();
    throw new Error(`DeepSeek 请求失败：${response.status} ${errorText}`);
  }

  const data = await response.json();
  return data.choices?.[0]?.message?.content ?? "";
}
```

这段代码的核心流程是：

- 把当前 `messages` 发给 DeepSeek。
- 如果 HTTP 状态不是成功，就把错误信息抛出来。
- 如果成功，就从 `choices[0].message.content` 里取出模型回复文本。

为什么是 `choices[0].message.content`？

因为 Chat Completions API 的响应通常长这样：

```json
{
  "choices": [
    {
      "message": {
        "role": "assistant",
        "content": "你好，我是 DeepSeek。"
      }
    }
  ]
}
```

## 6. 改造 index.ts

本章完整的 `src/index.ts`：

```ts
import { config } from "./config";
import type { ChatMessage } from "./types";

const input = process.argv.slice(2).join(" ").trim();

function addMsg(msg: ChatMessage) {
  messages.push(msg);
}

async function callDeepSeek(messages: ChatMessage[]): Promise<string> {
  const response = await fetch("https://api.deepseek.com/chat/completions", {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      Authorization: `Bearer ${config.DEEPSEEK_API_KEY}`,
    },
    body: JSON.stringify({
      model: config.DEEPSEEK_MODEL,
      messages,
    }),
  });

  if (!response.ok) {
    const errorText = await response.text();
    throw new Error(`DeepSeek 请求失败：${response.status} ${errorText}`);
  }

  const data = await response.json();
  return data.choices?.[0]?.message?.content ?? "";
}

// 缓存历史消息
const messages: ChatMessage[] = [
  {
    role: "system",
    content: "你是一个命令行agent!",
  },
];

if (!input) {
  console.error("请输入要执行的命令");
  process.exit(1);
}

if (!config.DEEPSEEK_API_KEY) {
  console.error("缺少 DEEPSEEK_API_KEY，请先在 .env 中配置。这个值不能提交到 Git。");
  process.exit(1);
}

addMsg({
  role: "user",
  content: input,
});

const assistantContent = await callDeepSeek(messages);

addMsg({
  role: "assistant",
  content: assistantContent,
});

console.log("当前消息列表：", messages);
```

现在第 03 章里的模拟回复：

```ts
content: "正在执行命令..."
```

已经变成真实模型回复：

```ts
const assistantContent = await callDeepSeek(messages);
```

## 7. 运行验证

运行：

```bash
npm run dev "你好啊，请用一句话介绍你自己"
```

如果配置正确，你会看到类似输出：

```text
当前消息列表： [
  { role: 'system', content: '你是一个命令行agent!' },
  { role: 'user', content: '你好啊，请用一句话介绍你自己' },
  { role: 'assistant', content: '你好，我是一个可以通过命令行与你交互的 AI 助手。' }
]
```

模型的具体回答每次可能不完全一样，这是正常的。

## 8. 常见错误

### 8.1 API Key 为空

如果看到：

```text
缺少 DEEPSEEK_API_KEY，请先在 .env 中配置。
```

说明 `.env` 没有配置，或者变量名写错了。

正确写法：

```env
DEEPSEEK_API_KEY=你的_api_key
DEEPSEEK_MODEL=deepseek-chat
```

### 8.2 401 Unauthorized

如果看到 `401`，通常是 API Key 不正确、过期，或者没有权限。

检查：

- `.env` 里的 Key 是否完整。
- Key 前后是否多了空格。
- 是否重新打开终端或重新运行命令。

### 8.3 模型名错误

如果返回模型不存在之类的错误，先使用：

```env
DEEPSEEK_MODEL=deepseek-chat
```

等跑通后，再换成其他模型。

### 8.4 fetch 报网络错误

如果 `fetch` 报网络错误，可能是网络无法访问 API，或者代理没有配置好。

这一类问题不是 TypeScript 代码错误，而是运行环境无法访问模型服务。

## 9. 这一章还没有做什么

本章只是为了跑通第一次真实模型调用，所以先不做这些事：

- 不封装 `LLMClient`。
- 不支持流式输出。
- 不处理工具调用。
- 不做重试和超时。
- 不兼容多个模型供应商。

这些都属于下一步的工程化内容。

下一章会把 `callDeepSeek(messages)` 抽到独立模块里，封装成统一的 `LLMClient` 接口。

## 10. 本章验收

完成本章后，你应该能：

- 理解为什么要先真实调用一次模型，再做封装。
- 知道 `fetch` 在这里的作用。
- 配置 `DEEPSEEK_API_KEY` 和 `DEEPSEEK_MODEL`。
- 用 `messages` 请求 DeepSeek Chat Completions API。
- 从 `choices[0].message.content` 读取模型回复。
- 把真实模型回复追加成 `assistant` 消息。
- 运行 `npm run dev "你好啊"` 得到真实模型返回。
